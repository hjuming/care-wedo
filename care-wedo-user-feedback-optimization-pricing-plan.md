# Care WEDO 用戶回饋優化與收費規劃

> 建立日期：2026-05-22  
> 依據：2026-05-19 實際用戶訪談回饋  
> 目標：先解決「好不好用」，再讓「要不要付費」與「資料安不安全」變得清楚可信。  
> 收費決策：採用版本 A，並於 2026-05-27 定稿單一家庭群組上限與 $30-250/月費用區間  
> 客服信箱：Care@wedopr.com

## 核心判斷

這次訪談不是要求增加一堆功能，而是指出 Care WEDO 的三個採用門檻：

1. 使用者要一眼知道怎麼用。
2. 使用者要知道何時需要付費、費用怎麼算。
3. 付費後，使用者要相信資料會被妥善保存，出問題有人可以聯絡。

因此開發順序應該是：

1. 先做高價值、低理解成本的「給醫生看」用藥總表。
2. 再補免費額度與升級提示。
3. 接著建立白話收費說明與本月費用預估。
4. 最後補資料安全、客服與正式付款流程。

## 收費方式草案

### 原則

1. 用戶第一層只看得懂「主要照護對象」與「共同協作者」。
2. 收費主軸以照護對象為主，因為資料保存成本主要跟照護對象數量相關。
3. 協作者費用要低，避免阻礙家人加入。
4. 所有加價動作都要先顯示預估月費，不自動加價。
5. Beta 期間先不收費，但介面要開始教育正式收費邏輯。

### 已採用版本 A：簡單好理解，優先推薦

| 項目 | 價格 | 說明 |
|---|---:|---|
| Free | $0/月 | 1 位照護對象、1 位登入者、每月 10 筆 AI 整理、可看未來提醒；保留最近 30 天資料，但不開放用戶查詢或觀看歷史資料 |
| 主要照護對象 | $30/人/月 | 照護圈升級後，每位照護對象可完整保存歷史紀錄，並有 100 筆/月整理額度 |
| 共同協作者 | +$10/人/月 | 可協助上傳、編輯、查看照護資料；主帳號不列入協作者費用 |
| 單一家庭群組上限 | $30-250/月 | 最多 4 位主要照護對象、5 位共同協作者、1 位主帳號 |

使用者看到的說明：

```text
免費可以先照顧 1 位家人。
Free 會保留最近 30 天資料，但歷史查詢與完整保存是付費功能。
如果你需要完整保存紀錄、邀請家人一起照護，升級後每位主要照護對象每月 $30。
每多 1 位共同協作者，加 $10。
主帳號不計入協作者費用。
每個家庭群組最多 4 位主要照護對象、5 位共同協作者；月費區間 $30-250。
```

案例：

| 情境 | 月費 |
|---|---:|
| 自己照顧媽媽，免費額度內使用 | $0 |
| 自己照顧媽媽，想保留完整歷史紀錄 | $30 |
| 自己 + 妹妹一起照顧媽媽 | $40 |
| 自己 + 妹妹 + 看護一起照顧媽媽 | $50 |
| 自己 + 妹妹一起照顧爸爸與媽媽 | $70 |
| 自己 + 妹妹 + 看護一起照顧爸爸與媽媽 | $80 |
| 4 位照護對象 + 5 位協作者 + 1 位主帳號 | $250 |

### 建議版本 B：完全按量計費，暫不推薦放前台

公式：

```text
月費 = 照護對象數 × $30 + 額外協作者數 × $10
```

優點：

- 對成本最直覺。
- 後台計算單純。

缺點：

- 使用者可能一開始就覺得「怎麼一加就要錢」。
- 不利於鼓勵家人加入。

建議：

- 可作為後台計算邏輯參考。
- 前台仍使用版本 A 的白話呈現。

## 權限規劃

| 角色 | 可以做 | 不可以做 |
|---|---|---|
| 付費者 / 照護圈管理者 | 新增與移除成員、建立照護對象、升級降級、查看完整紀錄 | 不可繞過付款限制 |
| 共同協作者 | 上傳照片、修改行程、更新用藥、查看資料 | 不可新增或移除成員、不可刪除照護對象、不可變更付款 |
| 照護對象本人 | 查看自己的今日照護、上傳資料、確認已完成 | 不應看到複雜管理與付款設定 |

### PR-CARE-P1-001：Medication Identity Normalization

目的：提高「給醫生看」用藥總表可信度，避免同一顆藥因 OCR 字串差異被列成多筆。

狀態：🟡 基礎完成（2026-05-29）。已補 `normalized_name`、`brand_name`、`generic_name`、`drug_code`、`dosage_text`、`identity_confidence`、`duplicate_candidate_ids`；LINE OCR exact duplicate 會更新同一筆，Web OCR pending review 會保留疑似重複候選，家人端會看到疑似重複提示，confirm API 只自動合併高信心 exact duplicate。正式藥碼資料源與完整合併管理尚未完成。

最小資料模型：

```ts
type MedicationIdentity = {
  rawName: string;
  normalizedName: string;
  brandName?: string;
  genericName?: string;
  drugCode?: string;
  dosageText?: string;
  sourceDocumentId: string;
  confidence: number;
  duplicateCandidateIds: string[];
};
```

驗收：

- [x] 同藥名不同空白、符號、大小寫時更新同一筆。
- [x] 商品名與學名同時出現時，儲存拆分欄位。
