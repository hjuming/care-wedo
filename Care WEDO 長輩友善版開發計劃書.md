## **1. 任務解讀**

| **項目** | **判斷**                                          |
| ------ | ----------------------------------------------- |
| 目標     | 讓 Care WEDO 從「功能介紹網站」轉為「長輩與家人一打開就能使用的照護工具」      |
| 核心對象   | 長輩、主要照顧者、家人協作者                                  |
| 主要場景   | 從 LINE 點開、看今日待辦、確認未來門診／檢查／領藥、查看吃藥               |
| 重要原則   | 已登入直接進用戶首頁；未登入才顯示 LINE 登入 CTA                   |
| 交付範圍   | 品牌色系、LINE 照顧管家回覆用語、操作介面、登入流程、首頁卡片、搜尋篩選、吃藥介面     |
| 風險     | 資訊過多、入口太深、老人家找不到功能、LINE WebView 登入狀態不穩、莫蘭迪色對比不足 |

# **2. 產品定位**

## **2.1 Care WEDO 的核心定位**

Care WEDO 不是一般醫療 App，也不是功能介紹網站。

它應該是：

一個長輩與家人打開 LINE 就能看懂的照護小幫手。

## **2.2 介面原則**

|**原則**|**說明**|
|---|---|
|首頁就是用戶首頁|已登入後，不再看到功能介紹首頁|
|今日優先|首頁只放最重要的門診、檢查、領藥、提醒|
|少即是多|首頁不放藥物明細、不放長篇說明|
|長輩可讀|字大、卡片清楚、操作按鈕明確|
|家人可查|未來、過去、醫院、診別、醫師用搜尋與篩選處理|
|功能說明移出首頁|關於我們、功能說明、服務介紹獨立成頁|

# **4. LINE 照顧管家回覆用語規範**

## **4.1 總原則**

|**原則**|**說明**|
|---|---|
|不重複問候|不要每一段都寫「親愛的爸爸／媽媽」|
|不塞藥名|領藥提醒只提醒領藥，不列複雜藥名|
|不過度解釋|長輩只需要知道「何時、何地、做什麼、帶什麼」|
|條列為主|一次一件事，一行一重點|
|重要資訊加粗|日期、時間、地點、號碼、健保卡|
|連結放最後|不打斷閱讀|
|語氣溫暖但不囉嗦|像家人提醒，不像客服公告|

# **6. 網站登入與首頁邏輯**

## **6.1 目標行為**

使用者從 LINE 開啟 Care WEDO 時：

|**狀態**|**進入頁面**|
|---|---|
|已登入|直接進入「今日照護首頁」|
|未登入|顯示清楚的 LINE 登入 CTA|
|登出後|回到未登入首頁|
|Session 過期|顯示 LINE 登入 CTA|
|LINE WebView 開啟|自動檢查登入狀態，不要進功能介紹頁|

---

## **6.2 路由設計**

|**路徑**|**用途**|
|---|---|
|`/`|智慧首頁：依登入狀態分流|
|`/login`|LINE 登入頁|
|`/app`|用戶今日照護首頁|
|`/app/future`|未來行程|
|`/app/search`|查詢與篩選|
|`/app/medications`|吃藥頁|
|`/app/family`|家人協作|
|`/about`|關於我們|
|`/features`|功能說明|
|`/help`|使用教學|
|`/settings`|設定|
|`/logout`|登出|

---

## **6.3 首頁分流邏輯**

```ts
/**
 * Care WEDO 首頁分流邏輯
 * 已登入：進 /app
 * 未登入：顯示 LINE 登入 CTA
 */
async function resolveHomeRoute() {
  const session = await supabase.auth.getSession()

  if (session?.data?.session) {
    return "/app"
  }

  return "/login"
}
```

## **6.4 登入狀態保存規則**

|**項目**|**規則**|
|---|---|
|登入方式|LINE Login / LIFF Login|
|Session 保存|Supabase Auth session persist|
|登入後導向|`/app`|
|下次開啟|若 session 有效，直接 `/app`|
|登出|清除 Supabase session 與本地狀態|
|不顯示|已登入狀態下不再顯示功能介紹首頁|

# **11. 吃藥介面設計**

## **11.1 首頁調整**

吃藥不顯示在今日照護首頁。

首頁只保留一個入口：

```text
[吃藥紀錄]
```

## **11.2 吃藥頁分組**

吃藥頁依時間區分：

1. 早
2. 中
3. 晚
4. 睡前
5. 其他

## **11.3 吃藥卡片**

```text
早上

白色圓形藥丸
飯後
每天一次

[吃了]
```

## **11.4 按鈕規則**

|**按鈕**|**處理**|
|---|---|
|吃了|保留|
|忘了|取消|
|未服用|不主動顯示，避免造成壓力|
|尚未記錄|系統狀態文字即可|

建議狀態：

```text
尚未記錄
已吃
由家人協助確認
```

# **14. 開發階段規劃**

## **Phase 1：品牌與 UI Token 整理**

|**項目**|**說明**|
|---|---|
|建立 Care WEDO 色票|寫入 design tokens|
|建立字級規範|手機版最低 18px|
|建立卡片樣式|門診、檢查、領藥、提醒|

驗收標準：

- 主按鈕明顯可點
- 莫蘭迪色只用於背景與輔助區，不影響辨識

---

## **Phase 2：登入分流與首頁重構**

|**項目**|**說明**|
|---|---|
|`/` 改為智慧分流|已登入進 `/app`，未登入進 `/login`|
|未登入首頁簡化|只保留 LINE 登入 CTA|
|功能介紹移出首頁|放到 `/features`|
|關於我們移出首頁|放到 `/about`|
|登入後自動記住狀態|除非登出，否則下次直接進 `/app`|

驗收標準：

- 從 LINE 開啟已登入帳號，直接看到今日照護首頁
- 不再出現功能介紹首頁擋住入口
- 登出後才回到登入頁
- LINE WebView 內不會反覆要求登入

---

## **Phase 3：今日照護首頁**

|**項目**|**說明**|
|---|---|
|建立今日卡片列表|門診、檢查、領藥、提醒|
|移除吃藥明細|吃藥只保留入口|
|今日空狀態|沒事項時提示查看未來行程|
|卡片排序|今天時間優先，緊急提醒置頂|

驗收標準：

- 長輩一開首頁就知道今天要做什麼
- 首頁不出現複雜藥名
- 首頁不需要捲太久就能看到重點

---

## **Phase 4：未來行程與查詢篩選**

|**項目**|**說明**|
|---|---|
|建立未來行程頁|本週、下週、本月、更久之後|
|建立查詢頁|關鍵字、日期區間、醫院、診別、醫師、類型|
|過去資料移至查詢|首頁不顯示過去紀錄|
|卡片共用元件|門診、檢查、領藥、提醒共用格式|

驗收標準：

- 可以查未來所有門診／檢查／領藥
- 可以依醫院、診別、醫師篩選
- 過去資料不干擾首頁

---

## **Phase 5：吃藥介面調整**

|**項目**|**說明**|
|---|---|
|分成早／中／晚／睡前／其他|依用藥時間顯示|
|保留吃了按鈕|快速記錄|
|移除忘了按鈕|避免長輩壓力與誤操作|
|顯示尚未記錄狀態|不用紅色警告造成焦慮|

驗收標準：

- 吃藥頁清楚分組
- 每張藥卡只有一個主要操作：「吃了」
- 首頁不再混入吃藥明細

## **Phase 6：LINE 照顧管家回覆優化**

|**項目**|**說明**|
|---|---|
|建立回覆模板|門診、檢查、領藥、提醒|
|OCR 摘要化|不輸出完整藥名與長篇說明|

驗收標準：
